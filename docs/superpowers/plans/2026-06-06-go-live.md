### 2. Keeper hasn't fired an end-to-end executeAllocation yet

Once tranche NAVs are non-zero AND the keeper's next hourly cycle picks up an approved verdict within the 24h TTL:

- Keeper reads RiskVerdictIssued + AllocationProposed.
- Fetches the allocation doc via the web proxy (seeded -> 200) or falls back to default-split.
- Calls executeAllocation(proposalId, senior[], mezz[], junior[]).
- Controller pulls USDC from idle buffer (or adapters via _ensureLiquidity), forceApproves each adapter, calls IYieldAdapter.deposit.
- AllocationExecuted event emitted.

Verify on Mantlescan that the keeper EOA sent the tx and that AllocationExecuted shows non-zero `seniorDeployed/mezzDeployed/juniorDeployed`.

Edge case: keeper boots more than 24h after Sentinel approves -> ProposalStale revert. Mitigation already in place: keeper runs hourly (well under 24h TTL).

### 3. ProposalTTL vs keeper cadence

ProposalTTL=24h. Keeper cycles hourly. As long as the Architect emits at least one proposal per 24h, the keeper picks it up before staleness. If demo demands shorter cycles, push the architect cadence down or extend TTL via `setProposalTTL`.

### 4. Adapter-name -> address map duplication

`agents/keeper/src/demo.ts` ADAPTER_ADDRESSES is hardcoded. If a new adapter ships on-chain, the map needs a manual edit. Not a go-live blocker, but flag for post-hackathon.
